# Kedama Design System — デザインルール

> このファイルは、Kedama Design System を構築・運用するときの **具体的なルール** を定めます。
> 人間が読んでも、AIに渡しても、このファイルだけで判断できる状態を目指しています。
>
> **デザイン原則**（なぜそうするか）は `docs/design-principles.md` を参照してください。
> このファイルは原則を「具体的にどうするか」に落とし込んだものです。

---

## 1. トークン設計ルール

### 1.1 2層分離（プリミティブとセマンティック）

デザイントークンは必ず **プリミティブ** と **セマンティック** の2層で管理する。

| 層 | 役割 | 例 | Figmaでの定義先 |
|---|---|---|---|
| プリミティブ | 値そのものを持つ。用途は持たない | `blue-500: #3B82F6` | Variables |
| セマンティック | 用途を意味する。値はプリミティブを参照する | `color.fg.default → gray-900` | Styles |

**禁止事項:**

- コンポーネント内で色・フォントサイズ等の値を直打ちしない。必ずセマンティックトークン経由で参照する
- セマンティックトークンに直接値（#xxxxxx等）を書かない。必ずプリミティブを参照する

**適用範囲:**

この2層構造はカラーだけでなく、以下すべてに適用する。

- カラー（color）
- タイポグラフィ（font-size, font-weight, line-height）
- スペーシング（spacing）
- シャドウ（shadow）
- 角丸（border-radius）

**なぜ:** 1か所変えれば全体に反映される。ダークモード対応やブランドカラー変更が容易になる。AIへの指示でも `color.fg.default を使って` と言えるため、出力精度が上がる。


### 1.2 トークンの命名規則

**プリミティブ:**

```
{カテゴリ}-{スケール}
```

- カラー: `gray-50`, `gray-100`, ... `gray-900`, `blue-500`, `red-500`
- フォントサイズ: `font-size-xs`, `font-size-sm`, `font-size-md`, `font-size-lg`, `font-size-xl`
- スペーシング: `spacing-1`(4px), `spacing-2`(8px), `spacing-3`(12px), ...

**セマンティック:**

```
{カテゴリ}.{要素}.{状態/バリアント}
```

- `color.fg.default` — テキストのデフォルト色
- `color.fg.muted` — 補助テキストの色
- `color.fg.inverse` — 反転背景上のテキスト色
- `color.bg.surface` — カードや領域の背景色
- `color.bg.page` — ページ全体の背景色
- `color.border.default` — 標準の境界線
- `color.accent.primary` — プライマリアクションの色
- `color.status.success` — 成功状態
- `color.status.warning` — 警告状態
- `color.status.error` — エラー状態
- `color.status.info` — 情報状態


### 1.3 タイポグラフィの命名

HTMLの見出しタグ（h1〜h6）に寄せた命名は使わない。デザインの視覚的な役割とHTMLのセマンティクスは別の軸であるため、用途ベースで命名する。

**使用するスタイル:**

| スタイル名 | 用途 |
|---|---|
| `heading-2xl` | ページタイトル |
| `heading-xl` | セクション見出し（大） |
| `heading-lg` | セクション見出し（中） |
| `heading-md` | セクション見出し（小） |
| `heading-sm` | カード見出し、サブセクション |
| `body-lg` | リード文、強調本文 |
| `body-md` | 本文（デフォルト） |
| `body-sm` | 補助テキスト、注釈 |
| `caption` | ラベル、メタ情報 |
| `overline` | カテゴリ表示、セクションラベル |

**注意:** 最初から全種を揃えない。実際に使うものから定義し、「既存スタイルで表現できない」と感じたときに追加する。使われないスタイルの量産は避ける。

**責任分界点:** Figmaのトークン命名はビジュアルの定義だけを担う。`heading-xl` が `<h2>` になるか `<h3>` になるかは文脈次第で、実装側が決める。ただし、コード側では適切なセマンティックHTMLとARIA属性を必ず付与する（原則2: Accessible）。ビジュアルの役割とHTMLのセマンティクスは、両方を満たす。

## 2. コンポーネント設計ルール

### 2.1 Auto Layout必須

Figmaでコンポーネントを作るとき、すべてAuto Layoutで組む。固定サイズのフレームは原則禁止。

**理由:**

- テキスト量の変動、ローカライズによるテキスト長の変化に自動対応できる
- AIツールが修正しやすい（固定サイズは意図が伝わらず崩れやすい）
- レスポンシブ対応の基盤になる

### 2.2 固定高さの原則と例外

固定高さは原則使わない。ただし以下のスクロールコンテナは例外として認める。

| コンポーネント | 固定高さの理由 |
|---|---|
| Sidebar | ページ全体の高さに固定し、内部をスクロールさせる |
| Dropdown Menu | 項目数が増えたとき最大高さで止めてスクロールさせる |
| Modal | 最大高さを超えたらコンテンツ部分をスクロールさせる |

**ルール:** 固定高さを使う場合はレイヤー名にコメントを書く（例: `[fixed-h: scroll container]`）。理由が記録されていれば、後から触る人やAIが意図的な固定であると判断できる。

### 2.3 レイヤー命名規則

`Frame 47` や `Rectangle 3` のようなデフォルト名を残さない。レイヤー名は設計意図を伝える手段である。

**命名構造:**

```
{カテゴリ}/{コンポーネント名}/{バリアント}/{状態}
```

例:

- `Actions/Button/Primary/Default`
- `Actions/Button/Primary/Hover`
- `Actions/Button/Secondary/Disabled`
- `Forms/Input/Default/Focus`
- `Feedback/Badge/Success`

**コンポーネント内部のレイヤー:**

- `_icon` — アイコン要素（`_` プレフィックスは内部要素を示す）
- `_label` — テキストラベル
- `_container` — 外枠コンテナ
- `_content` — コンテンツ領域

### 2.4 コンポーネントの必須状態

すべてのインタラクティブコンポーネントは、以下の状態を定義する。

- Default
- Hover
- Focus（キーボードフォーカス）
- Active / Pressed
- Disabled

すべてのデータ表示コンポーネントは、以下の3状態を定義する。

- Loading（読み込み中）
- Empty（データなし）
- Error（エラー）

**原則との関係:** これらの必須状態は、原則5（Simple）の「使うものだけ定義する」と一見衝突するが、原則の優先順位により Resilient > Simple で解決される。エラーハンドリングのために状態が増えることは、業務が止まるリスクを防ぐために許容する。

### 2.5 Props設計

- Propsの数は最小限にする。よく使う設定はデフォルト値で賄う
- booleanのPropsは肯定形で命名する（`isDisabled` ではなく `disabled`、`isLoading` ではなく `loading`）
- バリアントは実際にプロダクトで使うものだけ定義する

## 3. AI協業ルール

AIツール（Claude + Figma MCP等）と人間が同じファイルを触る環境で、衝突と品質低下を防ぐためのルール。これらはデザイン原則（Calm, Accessible, Resilient, Consistent, Simple）すべてを支えるプロセスルールであり、「ルールが言語化されていれば、人間でもAIでも正しく動ける」状態を維持するための仕組みである。

### 3.1 作業範囲の事前宣言

AIに指示を出すとき、必ず触ってよい範囲を明示する。範囲を書かなければ、AIは判断できずに想定外の箇所を変更する可能性がある。

**指示の書き方:**

```
作業範囲: Variables（カラーのプリミティブトークン）のみ
触れてはいけない範囲: Styles、コンポーネント
```

曖昧な指示の例（避ける）:
- ❌ 「色を整理して」
- ❌ 「このコンポーネントをいい感じに直して」

明確な指示の例:
- ✅ 「Variables > Color > gray のプリミティブトークン gray-50〜gray-900 を定義して」
- ✅ 「Button コンポーネントの Hover 状態の背景色を color.accent.primary-hover に変更して」

### 3.2 読み取り先行

いきなり変更を加えない。まず現状を確認し、レポートしてから変更に入る。

**手順:**

1. 「今このFigmaファイルに定義されているVariablesを一覧にして」
2. 一覧を確認し、差分を把握する
3. 「次は〇〇を変更して」と具体的に指示する

**なぜ:** 意図しない上書きを防ぐ。特にトークンの参照関係が壊れると影響範囲が大きい。

### 3.3 順序の徹底

デザインシステムの構築・変更は、必ず以下の順序で行う。上位の層が確定してから次に進む。

```
① プリミティブ（Variables） → ② セマンティック（Styles） → ③ コンポーネント
```

**禁止:** セマンティックトークンが未確定の状態でコンポーネントを作り始めること。ベースが壊れた状態でコンポーネントが量産されると、後から整合させるコストが膨大になる。

### 3.4 変更のレビュー

AIが行った変更は、必ず人間がレビューしてから確定とする。

**チェックポイント:**

- トークンの参照関係が正しいか（セマンティック → プリミティブ）
- 命名規則に従っているか
- Auto Layoutが適用されているか（固定サイズになっていないか）
- 不要なレイヤーやスタイルが追加されていないか
