# デザイン原則

> **驚きと感動で、明日が宝物になる体験を創る。**

業務システムにおける「驚きと感動」とは、派手な演出ではありません。
「こう動いてほしい」と思った瞬間にそう動くこと。気づかないうちにストレスが消えていること。
日々の業務が少しずつ心地よくなっていく——その積み重ねが「宝物」になる体験です。

---

## 原則の優先順位

原則同士が衝突した場合は、以下の順で優先します:

1. **Calm** — 穏やかであること
2. **Accessible** — 誰もが使えること
3. **Resilient** — しなやかであること
4. **Consistent** — 一貫していること
5. **Simple** — シンプルに保つこと

---

## 1. 穏やかであること（Calm）

業務UIは1日に何時間も向き合うもの。ユーザーの注意を最小限しか奪わず、必要な情報を穏やかに届けます。

- 彩度を抑えたカラーパレット。強い色はアクション・警告にのみ
- アニメーションは「状態変化の伝達」のためだけ
- ホワイトスペースを「機能」として使う——余白が情報の構造を伝える
- **Progressive Disclosure** — すべてを一度に見せない。1画面のCTAは原則1つ
- 周辺視野を活用する（通知バッジ、ステータスインジケーター）

**判断基準:** 「この要素は、ユーザーの注意を奪う価値があるか？」

---

## 2. 誰もが使えること（Accessible）

アクセシビリティは後付けではなく、設計の出発点。

- WCAG 2.2 AA 準拠がベースライン
- 色だけに依存しない情報伝達（アイコン・テキスト・パターンの併用）
- すべてのインタラクティブ要素はキーボード操作可能
- 適切な ARIA 属性とセマンティック HTML
- タイポグラフィ主導の階層（文字のサイズ・ウェイトで優先度を表現）

**判断基準:** 「色が見えなくても、マウスが使えなくても、この機能は使えるか？」

---

## 3. しなやかであること（Resilient）

壊れたとき使えなくなるUIは、業務を止めてしまう。

- 不正な Props やデータ欠損でもクラッシュせず、妥当なフォールバックを返す
- エラー状態には必ず「次に何をすればいいか」を提示する
- Loading / Empty / Error の3状態をすべてのデータ表示コンポーネントでデザインする
- **スマートデフォルト** — 何も設定しなくても使える初期状態

**判断基準:** 「このコンポーネントが壊れたとき、ユーザーは行き止まりにならないか？」

---

## 4. 一貫していること（Consistent）

同じ操作は、どの画面でも同じ見た目・同じ振る舞い。

- デザイントークンを必ず使い、ハードコードしない
- 同じ意味の操作には同じコンポーネント（例: 削除は常に danger ボタン）
- 命名規則を統一する

**判断基準:** 「別の画面で同じことをしたとき、ユーザーは迷わないか？」

---

## 5. シンプルに保つこと（Simple）

小規模チームで持続的に運用できることが大切。

- バリアントは実際に使うものだけ定義する（YAGNI）
- Props の数は最小限。よく使う設定はデフォルト値で賄う
- 人間の判断を増幅する。自動化するのは単純作業だけ

**判断基準:** 「これは本当に今必要か？もっとシンプルにできないか？」
